# Шпаргалка: что делаем, зачем и как работает «под капотом»

## 1) Какая задача

- **Цель**: предсказывать вероятность победы `p(win)` в матче ATP.
- **Почему вероятности**: нам важны качество вероятностей (logloss, калибровка), а не только бинарное решение.

## 2) Данные и целевая переменная

- **TML**: спортивные признаки и результат (winner/loser, rank, points, surface, даты).
- **ODS**: котировки букмекеров (PS/B365 и т.п.).
- **Склейка**:
  - агрегируем дату к `week` (начало недели),
  - нормализуем имена через `key_from_*`,
  - задаём упорядоченную пару игроков `p1/p2` (`min/max` по ключу),
  - join по `week + p1 + p2`.
- **Target**: `y_p1_win = 1`, если победитель совпал с `p1`, иначе `0`.

## 3) Baseline букмекера (`ps_imp_p1`)

- Берём odds и переводим в implied probability:
  - `p1_raw = 1/odds_p1`, `p2_raw = 1/odds_p2`
  - нормализация от маржи: `ps_imp_p1 = p1_raw / (p1_raw + p2_raw)`
- Почему baseline сильный:
  - букмекерская линия агрегирует много сигналов (форма, травмы, покрытие, мотивация, инсайд).

## 4) Logistic Regression: `with_odds` / `no_odds`

### `logreg_with_odds`

- Features:
  - числовые: `ps_imp_p1`, `b365_imp_p1` (если доступны)
  - категориальные: `surface` (one-hot)
- Под капотом:
  - `ColumnTransformer`:
    - числовые → median impute + scaling
    - категориальные → most_frequent + one-hot
  - `LogisticRegression(saga, l2)` учит веса для `P(y=1|x)`.

### `logreg_no_odds`

- Features:
  - `rank_diff_p1_minus_p2`
  - `rank_points_diff_p1_minus_p2`
  - `surface`
- Это baseline “sports-only” (без букмекера).

## 5) ELO overall + ELO surface

### Зачем

- ELO — онлайн-рейтинг силы, который “накапливает” прошлую информацию и часто даёт сильный сигнал без букмекера.

### Как считаем

- Для каждого игрока храним:
  - `ELO_overall`
  - `ELO_surface` (отдельно для каждого покрытия)
- В матче используем значения **до обновления**:
  - `elo_diff = elo_p1 - elo_p2`
  - `elo_surface_diff = elo_surface_p1 - elo_surface_p2`

### Формулы

- Ожидаемая победа:
  - `E = 1 / (1 + 10^((R2 - R1)/400))`
- Апдейт после матча (K-factor):
  - `R1 <- R1 + K*(S1 - E1)`
  - `R2 <- R2 + K*(S2 - E2)`
- В ноутбуке: `K=32`, `init=1500`.

### `logreg_no_odds_elo`

- Это `no_odds` + (`elo_diff`, `elo_surface_diff`).

## 6) Walk-forward validation (time series)

### Зачем

- Исключить утечку будущего и проверить устойчивость выводов во времени.

### Как

- Cutoffs: годовые (`2019-01-01 … 2025-01-01`)
- На каждом cutoff:
  - train: `week < cutoff`
  - test: `cutoff <= week < cutoff + 365 days`

## 7) Метрики

- **LogLoss**: штраф за уверенные ошибки, главная метрика для вероятностей.
- **ROC AUC**: качество ранжирования (инвариантно к порогу).
- **Accuracy@0.5**: грубая метрика, сильно зависит от порога.

## 8) Калибровка

- **Reliability diagram**: predicted probability vs empirical frequency.
- **Brier score**: MSE по вероятностям.
- Важно: хорошая AUC не гарантирует хорошую калибровку.

## 9) Bootstrap CI (что и зачем)

### Цель

- Получить доверительные интервалы для разниц:
  - `Δlogloss` и `ΔAUC` между моделями.

### Как правильно

- Bootstrap делается на **одном и том же тест-наборе**:
  - ресэмплим тестовые матчи с возвращением,
  - пересчитываем метрики,
  - строим распределение `Δ`, берём квантиль 2.5% и 97.5% (95% CI).

### Типовые сравнения

- `baseline_ps_only` vs `logreg_with_odds`
- `logreg_no_odds` vs `logreg_no_odds_elo`

### Важный нюанс

- Чтобы bootstrap был корректным, нужно сохранять **вектор предсказаний** `p_pred` на тесте для каждой модели (а не только агрегированные метрики).
